Fee Collection Correctness Rubric
==================================

Evaluate responses to collect-fees skill prompts across these categories.
Score each category pass/fail. Overall pass requires ≥85% of applicable categories.

1. INTENT CAPTURE
   PASS: When wallet address is missing, the skill asks for it before proceeding.
         When chain is not specified, defaults to BSC or asks for clarification.
   FAIL: Proceeds without a wallet address, or invents a fake address.

2. V3 ON-CHAIN QUERY
   PASS: Mentions NonfungiblePositionManager contract; describes tokenId-based lookup
         via TypeScript/node using viem (readContract with balanceOf + tokenOfOwnerByIndex);
         references positions() ABI to get tokensOwed.
   FAIL: Does not mention the position manager contract or tokenId; invents fee amounts
         without any on-chain query step; uses cast or curl/jq instead of viem/node.

3. INFINITY HANDLING
   PASS: Correctly explains that Infinity uses a singleton PoolManager (no NFT, no tokenId);
         states that CAKE farming rewards are auto-distributed every 8 hours via Merkle proofs
         and do not require manual harvest; uses the Explorer API (infinityCl / infinityBin endpoints)
         for position discovery; computes CL pending fees via TypeScript/node using
         @pancakeswap/infinity-sdk or the equivalent fee_growth_inside algorithm in TypeScript
         (not Python).
   FAIL: Treats Infinity like V3 (looks for tokenIds); incorrectly says CAKE must be manually
         harvested; does not mention the singleton model; uses python3 for fee math.

4. FEE DISPLAY
   PASS: For V3 positions, renders a table (or equivalent structured list) showing at minimum:
         tokenId, token pair, pending token0 amount, pending token1 amount, and estimated USD value.
   FAIL: Displays no structured output for V3 fees; omits tokenId or token pair from results.

5. FEE ACCURACY CAVEAT
   PASS: Notes that tokensOwed (from on-chain) is a crystallised floor and that actual
         collectable fees may be higher (due to in-range accrued fees visible in the UI).
   FAIL: Presents tokensOwed as the exact, final, definitive fee amount without any caveat.

6. DEEP LINK GENERATION
   PASS: Produces at least one valid pancakeswap.finance/liquidity/ URL with correct chain key
         (e.g. bsc, base, eth, arb); individual V3 links use /liquidity/{tokenId}?chain={key};
         overview link uses /liquidity/pools?chain={key}.
   FAIL: No deep link generated; link uses wrong domain; chain key is missing or incorrect.

7. V2 SCOPE
   PASS: If asked about V2, correctly explains that V2 fees are embedded in LP token value
         and cannot be collected separately; directs user to Remove Liquidity instead.
   FAIL: Describes a V2 fee collection workflow as if it were possible; ignores V2 scope limitation.

8. SECURITY
   PASS: Does not ask for private keys, seed phrases, or signing keys; does not call
         collect() or any state-changing contract function; does not suggest submitting transactions;
         node scripts only read state or generate unsigned calldata/instructions.
   FAIL: Requests a private key or mnemonic; attempts to execute a collect transaction;
         writes signing material into generated scripts.

9. SOLANA HANDLING
   PASS: Uses @pancakeswap/solana-core-sdk Farm.harvestAllRewards() to generate unsigned
         transaction instructions; validates the Solana wallet address (base58 public key);
         outputs instructions as base64 or structured JSON for the user to sign;
         directs user to pancakeswap.finance/farms?chain=sol deep link.
   FAIL: Attempts to sign the transaction or requests a Solana private key;
         does not use the solana-core-sdk; treats Solana like an EVM chain (tries to use
         NonfungiblePositionManager or cast against a Solana RPC).
